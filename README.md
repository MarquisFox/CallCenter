# Система анализа звонков – микросервисная архитектура

## 1. Общее описание

Система предназначена для автоматической обработки записей телефонных разговоров: распознавания речи, анализа тональности, оценки качества диалога по заданному чек-листу (с использованием GigaChat), накопления статистики по менеджерам и предоставления отчётов через HTTP API.

Компоненты упакованы в Docker-контейнеры, взаимодействуют через сеть `backend` и обмениваются сообщениями через Apache Kafka. Основное хранилище данных – PostgreSQL, для кеширования и хранения состояния ретраев используется Redis.

---

## 2. Принятые архитектурные решения

### 2.1. Микросервисная архитектура

Каждый этап обработки звонка (приём метаданных, STT, тональность, GigaChat‑анализ, статистика) выделен в отдельный микросервис. Это позволяет:

- независимо масштабировать компоненты (STT требует GPU, тональность – CPU, статистика – память);
- изолировать сбои – проблема в одном сервисе не останавливает всю систему;
- переиспользовать сервисы (например, тональность можно заменить на другую модель без изменения остальных).

### 2.2. Брокер сообщений Kafka и паттерн «production ready queue»

Все асинхронные взаимодействия построены на Apache Kafka. Это даёт:

- надёжную доставку сообщений;
- возможность перечитывать поток с любой позиции;
- независимую работу сервисов (каждый потребляет в своём темпе).

Для промышленной эксплуатации реализован паттерн **Production Ready Queue**:

- Смещение (offset) фиксируется **вручную** только после полной успешной обработки сообщения.
- При ошибке сообщение не коммитится, и сервис повторяет попытку с **экспоненциальной задержкой** (backoff).
- Количество попыток ограничено; после исчерпания сообщение отправляется в **Dead Letter Queue (DLQ)** с сохранением исходных данных и текста ошибки. Это позволяет не блокировать очередь и даёт возможность администратору проанализировать проблему позже.

Счётчики ретраев хранятся в **Redis** – это общее in‑memory хранилище, которое сохраняет информацию о попытках даже при перезапуске сервиса.

### 2.3. Хранилище данных: нормализованная PostgreSQL

Основное хранилище – PostgreSQL, таблицы приведены к третьей нормальной форме:

- сущности «менеджер», «звонок», «тональность», «статус», «пункт чек-листа» имеют собственные первичные ключи;
- связи реализованы через внешние ключи (`call.manager_id → manager.id`, `call_checklist_result.call_id → call.id` и т.д.);
- агрегированная статистика менеджера (`manager_stats`) обновляется транзакционно при завершении звонка.

Нормализация исключает дублирование, упрощает расширение чек-листа и гарантирует целостность данных.

### 2.4. Кеширование в Redis

Часто запрашиваемые отчёты (дашборд, список менеджеров с процентом ошибок, детали менеджера) кешируются в Redis с помощью Spring Cache (`@Cacheable`). Это снижает нагрузку на PostgreSQL в десятки раз и обеспечивает быстрые ответы API (миллисекунды вместо сотен миллисекунд). Инвалидация кеша происходит при обновлении статистики менеджера (`@CacheEvict`).

### 2.5. API Gateway

Все внешние запросы (от CRM и от фронта) проходят через сервис `BitrixAdapter` – единственную точку входа в систему. Он:

- принимает `POST /call-events`, валидирует и нормализует данные, отправляет сообщение в Kafka;
- проксирует GET-запросы статистики на внутренний `Statistics Service` через реактивный `WebClient`.
- Порт 8081 – единственный, который открыт наружу. PostgreSQL, Kafka, Redis и сам `Statistics Service` не имеют публичных портов и доступны только внутри Docker-сети.



## 3. Схема потоков данных

1. **CRM → BitrixAdapter** → Kafka topic `call.registration`.
2. **STT‑сервис** потребляет `call.registration`, скачивает аудио по presigned URL, транскрибирует (Whisper), отправляет в `transcription.completed`.
3. **Тональность и Gigachat** читают `transcription.completed`.
4. **Тональность** (Dostoevsky) публикует результат в `tonality.analysis.output`.
5. **Gigachat** (через промт с чек-листом) публикует рейтинг, errorRate и пункты чек-листа в `recommendation.analysis.output`.
6. **Statistics Service** потребляет `call.registration` (создаёт звонок и менеджера), `tonality.analysis.output` (обновляет тональность), `recommendation.analysis.output` (обновляет рейтинг, ошибки, пункты чек-листа). После получения обоих результатов пересчитывает статистику менеджера.
7. **Frontend** запрашивает отчёты через BitrixAdapter (прокси в Statistics Service).

## 4. API (через шлюз, порт 8081)

| Метод | Эндпоинт | Назначение |
|-------|----------|-------------|
| `POST` | `/api/v1/call-events` | Добавление звонка (callId, managerId, fileUrl и др.). |
| `GET` | `/api/v1/calls/get-all-calls?page=0&size=20` | Пагинированный список звонков. |
| `GET` | `/api/v1/dashboard/aggregated?startDate=...&endDate=...&managerIds=...` | Агрегаты для дашборда (callsByDay, durationByDay, callsByManager, errorRateByManager). |
| `GET` | `/api/v1/managers/get-all-managers-with-error-rate` | Список менеджеров с процентом звонков, содержащих ошибки. |
| `GET` | `/api/v1/managers/get-manager-details?id=...&startDate=...&endDate=...&tonalityIds=...` | Детальная информация о менеджере (средний рейтинг, errorRate, список ошибок, звонки). |

OpenAPI спецификация доступна после запуска: `http://localhost:8081/swagger-ui.html`.
